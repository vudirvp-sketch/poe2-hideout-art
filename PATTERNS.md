# PATTERNS.md — закономерности композиции .hideout изображений

> Source of truth по результатам анализа 7 файлов (6 PoE1 + 1 PoE2).
> Все цифры — из `download/spatial_stats.json` и `download/per_type_stats.json`.
> Не редактировать без перезапуска анализа.

## 1. Формат файла

### 1.1 Два варианта синтаксиса

| Format | Header | Placement syntax | Encoding |
|---|---|---|---|
| **New (JSON)** | `"version": 1, "language": "...", "hideout_name": "...", "hideout_hash": N` | `"Name": { "hash":N, "x":N, "y":N, "r":N, "fv":N }` | UTF-8 (с/без BOM) |
| **Old (text)** | `Language = "..."`, `Hideout Name = "..."`, `Hideout Hash = N` | `Name = { Hash=N, X=N, Y=N, Rot=N, Flip=N, Var=N }` | UTF-16-LE (BOM) |

### 1.2 Duplicate keys = multiple placements

PoE .hideout JSON использует **один и тот же ключ多次** для представления нескольких placements одного и того же doodad'а.

> ⚠️ **CRITICAL**: стандартный `json.load()` молча оставляет только последнее вхождение каждого ключа. Это даёт счёт 21 вместо 450, 42 вместо 568 и т.д. Всегда использовать regex-парсер `scripts/parse_hideout.py`.

### 1.3 Поля placement'а

| New field | Old field | Тип | Описание |
|---|---|---|---|
| `hash` | `Hash` | uint32 | Идентификатор doodad-типа. Одинаковое имя в PoE1 и PoE2 → разные хеши. |
| `x`, `y` | `X`, `Y` | int | Координаты на сетке убежища. **Всегда целые.** |
| `r` | `Rot` | uint16 | Поворот: `0..65535` = `0..360°`. Фиксированная точка 16-бит. `65536 = 360°`, `16384 = 90°`, `32768 = 180°`, `49152 = 270°`. |
| `fv` | `Flip`+`Var` | uint8 | `(flip << 7) \| var`. `Var` = вариант модели (0..18 встречаются), `Flip` = зеркало (0/1). |

### 1.4 Сетка координат

- Y растёт **вниз** (экранная система координат, не математическая).
- X растёт вправо.
- Типичный bbox изображения: 112..235 × 220..360 единиц.
- Пиксельный pitch ≈ 1 единица = 1 клетка.

## 2. Структура изображения

### 2.1 Количество placements

| Параметр | Min | Median | Max |
|---|---|---|---|
| Всего placements | 450 | 624 | 764 |
| Уникальных doodad-типов | 21 | 34 | 47 |
| Aspect (W/H) | 0.47 | 0.74 | 0.91 (только poe2 > 1.0) |

**5 из 7 файлов — portrait** (aspect < 0.75). Только `poe2.hideout` — landscape (это сцена, не портрет).

### 2.2 Координаты — integer grid, hand-placed

- **Все** координаты `x, y` — целые числа.
- Nearest-neighbor distance: **median = √2 ≈ 1.414** (диагональная соседняя клетка).
- distinct x = 99..168, distinct y = 99..168 — то есть из 500-700 точек лишь ~25% уникальных строк/столбцов.
- **НЕ регулярная сетка** и **НЕ hex packing**: точки стоят на целочисленной решётке, но выбираются вручную (часть клеток занята, часть пуста).

### 2.3 Алгоритм размещения (универсальный)

```
Для каждого "пикселя" картинки:
  1. Выбрать целевую (x, y) на integer grid (получено из исходного image downsampling).
  2. Выбрать doodad-тип по роли пикселя (fill / stroke / outline / accent).
  3. Выбрать rotation r из квантованного набора (см. §3).
  4. Выбрать variant fv (var + flip<<7).
  5. Добавить запись в "doodads".
```

Точки **всегда** на integer grid → не нужно хранить суб-pixel координаты. Плотность регулируется пропуском клеток.

### 3.4 Назначение ротации

- **Contour following** (Cat, CatGirl): hue градиент вдоль силуэта (голова→хвост).
- **Edge-normal encoding** (poe2): разные углы у левого/правого края.
- **Zone segmentation** (Nayuta, cross): разные регионы — разные угловые кластеры.
- **Default-orientation tiles** (Pikachu): почти без вращения.

## 5. Канонический алгоритм img2hideout (вывод из анализа)

```
INPUT: source image (photo / portrait / scene)
OUTPUT: .hideout file (new JSON format)

1. Downsample source до целевой плотности (target N=500-700 placements).
   - Размер canvas: ~200×260 единиц (portrait) или 235×258 (landscape).
   - Crop / letterbox под aspect.

2. Классифицировать каждый "пиксель" downsapled image в одну из ролей:
   - FILL  → внутренние точки силуэта (>= 2同类-соседей в downsapled grid)
   - OUTLINE → точки на границе (1 сосед внутри, остальные снаружи)
   - STROKE → точки на тонких линиях (волосы, текстуры)
   - ACCENT → редкие цветовые акценты

3. Подобрать doodad-тип под роль:
   - FILL  → 1 канонический fill-тип (PoE1: 火山卵石; PoE2: 洞穴石化像)
   - STROKE → 1-3 канонических stroke-типа (PoE1: 大理石板/營地地毯/飄雪; PoE2: 大理石長凳)
   - OUTLINE → мелкие типы (寶石袋, 冰湖, 廢棄教堂)
   - ACCENT → редкие акцентные (藍色光束, 新鮮人體器官)

4. Вариация за счёт ротации (r):
   - Квантовать до 11.25° или 22.5° (snap).
   - Для FILL/STROKE: предпочтительные углы 0°, 315°, 337.5°, 348.75° (back-quadrant).
   - Для OUTLINE: подбирать по локальному направлению контура (edge-normal).
   - Добавлять небольшую случайность (±5°) для разрушения моарé.

5. Вариация за счёт fv (variant):
   - Если у doodad'а есть 4+ variant'ов — равномерно распределять.
   - flip=1 ставить ~30% placements для асимметрии.

6. Не дублировать (x, y) для одного doodad-типа.
   Можно ставить несколько разных типов в одну клетку (overlap разрешён).

7. Не использовать functional singleton'ов (倉庫/傳送點/工藝台) для картинки —
   они требуют реальных позиций в убежище.

8. Сохранить в new JSON format с duplicate keys.
```

## 6. Анти-паттерны (что НЕ делать)

| Анти-паттерн | Почему плохо | Альтернатива |
|---|---|---|
| Регулярная сетка с равным шагом | Выглядит механически, моарé | Integer grid с пропусками клеток |
| Все точки под одним углом | Плоско, нет direction | Квантовать + варьировать по зонам |
| Один doodad-тип на всё изображение | Нет visual hierarchy | 1 FILL + 2-3 STROKE + 5+ OUTLINE/ACCENT |
| Bezier/thick_ring/crosshatch примитивы | Не встречаются в референсах | Curved contour strokes (no hatching) |
| Преобразование PNG→hideout "пиксель-в-пиксель" | Слишком много placements, лагает игра | Downsample до 500-700 placements |
| `json.load()` для чтения готовых .hideout | Теряет duplicate keys → wrong counts | Regex parser в `scripts/parse_hideout.py` |

## 7. Референсные значения для генерации

### 7.1 Размер canvas

| Orient | W (units) | H (units) | Placements |
|---|---|---|---|
| Portrait | 110–200 | 230–360 | 500–700 |
| Landscape | 235 | 258 | 400–500 |

### 7.2 Палитра ролей (target distribution)

```
FILL:     1 type, 30-60% of placements, NN-same ≈ 1.4
STROKE:   1-3 types, 10-25% each, NN-same ≈ 2-5
OUTLINE:  3-8 types, 1-10% each, NN-same variable
ACCENT:   5-15 types, 1-5% each, NN-same > 5
SINGLETON: 5-20 functional, 1 each (не часть картинки)
```

### 7.3 Rotation palette

```
Default: 0° (r=0)
Secondary: 315° (r=57344) — universal second angle
Tertiary: 337.5° (r=61440), 348.75° (r=63488), 303.75° (r=55296)
PoE2-specific: 311.6° (r=56721) — фактическая доминанта в poe2.hideout
Quantum: 11.25° (4096 в r)
```

## 8. Что требует дополнительных референсов

- **Цвет декор** как perceived in-game: этот анализ только по позициям/ротациям, без учёта реальных текстур и цветов. Нужен скриншот в игре каждого топ-doodad'а, чтобы построить "color palette".
- **PoE2 hides отличия от PoE1**: только 1 PoE2-файл в корпусе. Нужны ещё 2-3 PoE2 изображения для устойчивых выводов.
- **Edge-normal rotation encoding**: гипотеза, основанная на 1 файле (poe2). Требует верификации на дополнительных примерах.
- **Варианты variant (fv)**: какие значения 0..18 соответствуют каким визуальным стилям — нужно инспектировать в игре.
